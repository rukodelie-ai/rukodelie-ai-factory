## 4. АРХИТЕКТУРА СИСТЕМЫ

### MVP-архитектура (текущий этап)

```
КЛИЕНТ (Telegram)
      │ сообщение / фото / кнопка
      ▼
TELEGRAM BOT (Python + aiogram)
      │ webhook
      ▼
MESSAGE ROUTER
  /start → приветствие
  /catalog → категории
  текст/фото → AI Seller Handler
      │
      ▼
AI SELLER CORE (Claude API)
  model:       claude-sonnet-4-6
  temperature: 0.3
  max_tokens:  1024
  system:      SYSTEM_PROMPT.md (<system_prompt> блок)
               + <customer_context> (если есть история)
               + каталог (только published товары)
  messages:    история диалога (последние 20 сообщений)
      │                    │
      ▼                    ▼
PRODUCT DATABASE      ORDER MANAGEMENT
(Google Sheets)       (Google Sheets)
  лист: catalog         лист: orders
  лист: settings        лист: analytics
```

### Цикл заказа (9 шагов)

```
new → confirmed → paid → assembling → shipped → delivered → closed
```

| Статус | Действие AI |
|--------|------------|
| `new` | "Заказ принят, менеджер свяжется в 30 мин" |
| `confirmed` | "Заказ подтверждён, ожидает оплаты" |
| `paid` | "Оплата получена ✅ Начинаем сборку!" |
| `assembling` | "Собираем ваш заказ 📦" |
| `shipped` | "Заказ отправлен 🚀 Трек: [номер]" |
| `delivered` | "Всё получили? 😊 [персональная рекомендация]" |
| `closed` | Маркетинговый сценарий через 7 дней |

### Память клиента

В system prompt перед каталогом передаётся XML-блок:

```xml
<customer_context>
  <name>Айгерим</name>
  <total_orders>7</total_orders>
  <last_order_date>2026-05-20</last_order_date>
  <favorite_categories>Стразы Swarovski, Бисер Preciosa</favorite_categories>
  <last_products>Стразы ss16 Crystal AB × 200 шт</last_products>
  <preferred_colors>Crystal AB, коралловый</preferred_colors>
  <order_frequency>раз в 2–3 недели</order_frequency>
  <segment>vip</segment>
</customer_context>
```

Сегменты: `new` (0 заказов) / `regular` (1–5) / `loyal` (6–15) / `vip` (16+)

> Этот блок сегодня пустой/статичный. **AI Memory MVP** (утверждён 2026-07-08,
> см. раздел 14) начнёт заполнять его реальными данными из истории покупок 1С —
> без изменения структуры блока или SYSTEM_PROMPT.md.

## 6. КЛЮЧЕВЫЕ РЕШЕНИЯ

| # | Решение | Обоснование |
|---|---------|-------------|
| 1 | **AI Seller = персона "Алина"**, консультант 5 лет | Человекоподобная персона даёт конверсию выше безымянного бота |
| 2 | **System prompt в тегах `<system_prompt>`** | Документация команды отделена от промпта API; безопасно обновлять |
| 3 | **claude-sonnet-4-6, temperature=0.3** | Точное следование правилам + естественная речь; 0.0 звучит роботизированно |
| 4 | **Язык определяется автоматически** | Вопрос о языке ломает первый контакт |
| 5 | **Хранилище заказов — Google Sheets (MVP)** | Нет деплоя БД; владелец видит без технических знаний; заменяется в v2 |
| 6 | **Память клиента через XML-блок** | Claude хорошо парсит структурированный XML; опционален — работает без него |
| 7 | **Сегменты: new/regular/loyal/vip** | 4 уровня достаточно для персонализации |
| 8 | **sale_mode = both** для страз и лент | Продаём и поштучно и упаковкой; товар не снимается при stock < pack_qty |
| 9 | **Маркетинговый сценарий: 1 за диалог** | Агрессивный маркетинг разрушает доверие |
| 10 | **Передача менеджеру < 20% обращений** | Менеджер — исключение, не правило |
| 11 | **Только `status = published` видно AI** | Черновики и снятые товары не существуют для клиента |
| 12 | **Каталог передаётся в system prompt** | Не через RAG/embeddings (MVP); достаточно для 50–200 товаров |
| 13 | **RAM dict для истории диалогов** | SQLite добавляет 57% кода при нулевом преимуществе до 100 сессий; при рестарте история сбрасывается — допустимо для MVP |
| 14 | **catalog/ пакет вместо плоского файла** | OData-ready: csv_provider.py + __init__.py; будущий odata_provider.py добавится без изменений интерфейса бота |
| 15 | **Диагностика перед интеграцией** | odata_probe.py и odata_sales_probe.py созданы ДО odata_provider.py; решение принимается на основе данных, не гипотез |
| 16 | **OData: ждём ответов Артёма** | 1077 sales сущностей найдено; odata_provider.py НЕ создаётся до ответов специалиста 1С на 5 вопросов |
| 17 | **Правило: без проектирования ради проектирования** | Документы и архитектуры создаются только когда есть конкретный реализуемый результат, не ради порядка |
| 18 | **Правило: одно решение — один завершённый результат** | Каждое управленческое решение заканчивается конкретным файлом/функцией/процессом — не планом |
| 19 | **Правило: оценка по прибыли 30–90 дней** | Новые функции и агенты оцениваются только по влиянию на прибыль в горизонте 30–90 дней |
| 20 | **Ошибка уведомления менеджера не должна терять заказ** | `save_order()` выполняется и логируется ДО попытки уведомления; `_notify_manager()` изолирован в try/except — сбой Telegram-доставки не откатывает и не блокирует уже сохранённый заказ |
| 21 | **ADMIN_CHAT_ID — реальный chat_id, не токен** | В отличие от TELEGRAM_BOT_TOKEN/ANTHROPIC_API_KEY, chat_id не секрет — логируется открыто для диагностики; ошибка «chat not found» означает неверный/плейсхолдерный chat_id или что бот ни разу не получал сообщение от этого аккаунта |
| 22 | **Телефон — основной бизнес-идентификатор клиента** | Единый идентификатор для всех каналов (Telegram/WhatsApp/сайт/приложение); внутренний ID контрагента в 1С — техническая деталь реализации, не бизнес-сущность |
| 23 | **1С = System of Record, AI Factory = System of Intelligence** | 1С хранит факты покупок (документы, суммы, даты); AI Factory вычисляет выводы (сегмент, предпочтения, контекст общения) — не дублирует и не подменяет учётную систему |
| 24 | **Первая практическая OData-интеграция — AI Memory MVP, не полный каталог** | Owner Review 2026-07-08: вместо полного перехода каталога/остатков на OData — узкая интеграция истории покупок клиента по телефону; полный каталог отложен на неопределённый срок |

| 35 | **Пилотная категория Master Product Database — только «Пряжа»** | Полный цикл (RAW → аудит → V2 → AI Product Knowledge) сначала проверяется на одной категории; масштабирование на остальные категории — только после успеха пилота |
| 36 | **Цветовые варианты товара доступны через OData по связке `Номенклатура → Характеристика_Key`** | Точечная валидация на Puffy Fine Alize (PHASE 1.5) выявила отсутствие цветов в PHASE 1 RAW (FAIL); найдена рабочая схема: `AccumulationRegister_СвободныеОстатки_RecordType` / `InformationRegister_ЦеныНоменклатуры_RecordType` / `Catalog_iq_ИзображенияХарактеристик`, все три фильтруются по `Номенклатура_Key` и несут `Характеристика_Key` — дают цвет, код, название, фото, остаток по складу и цену на каждый вариант |
| 37 | **Ложная гипотеза «проблема на стороне публикации OData 1С» — выдвинута и опровергнута** | Ошибка `AUTOORDER` при фильтрации регистров по `Номенклатура_Key`/`Характеристика_Key`/`Склад_Key` изначально выглядела как серверный сбой; контрольная проверка тем же Python `urllib`-кодом, что работал 2026-07-09, прошла без ошибок — проблема была не в 1С |

## 11. ВАЖНЫЕ ПРАВИЛА ПРОЕКТА

### Правила AI Seller (неизменяемые)

1. **Только published товары.** `status ≠ published` → товар не существует для клиента.
2. **Автоопределение языка.** Не спрашивать клиента о языке.
3. **Один вопрос за раз.** Никогда не задавать несколько уточнений подряд.
4. **Всегда называть цену и наличие.** Клиент не должен спрашивать дважды.
5. **Не давать скидок.** Только из прайса. Оптовые условия — менеджер.
6. **Не называть реквизиты.** Оплата только через менеджера.
7. **Не обсуждать конкурентов.** Ни хорошо, ни плохо.
8. **Эскалация < 20% обращений.** Менеджер — исключение, не правило.
9. **Маркетинг: 1 сценарий за диалог.** Не применять при жалобах и оформлении.
10. **Карточка товара — строго по шаблону.** Не длиннее 8 строк.

### Правила работы с каталогом

- `sale_mode = both`: продаём и поштучно, и упаковками. Товар НЕ снимается если stock < pack_qty.
- `stock` хранится в базовых единицах (`base_unit`): стразы — в штуках, ленты — в метрах.
- Расчёт полных упаковок: `FLOOR(stock / pack_qty)`. Остаток = `stock MOD pack_qty`.
- Пряжа: всегда называть состав, метраж, рекомендуемые спицы/крючок.
- Стразы: всегда называть размер (ss), способ крепления, цену за шт. И за упаковку.
- Бегунки/молнии: обязательно уточнять номер (N3/N5/N8) — должны совпадать.

### Правила разработки фабрики

- **Automation First:** любой повторяющийся процесс автоматизируется.
- **Single Source of Truth:** catalog_v1.csv — единственный источник данных о товарах.
- **Data-Driven:** каждое решение опирается на данные (Analytics Agent в Этапе 2).
- **Owner-in-the-Loop:** владелец получает недельные дайджесты, участвует только стратегически.
- **Quality over Speed:** лучше меньше, но правильно.

### Управленческие принципы (решения владельца, утверждены 2026-07)

- **Одно решение — один завершённый результат.** Не план, не документ — готовый работающий результат.
- **Без проектирования ради проектирования.** Новые документы и архитектуры — только под конкретный результат.
- **Правило 30–90 дней.** Любая новая функция или агент оцениваются по прибыльному эффекту в течение 30–90 дней.
- **AI Growth Engine.** Стратегическое направление утверждено; включает Video Agent + Marketing; первый результат — MVP 10 Reels.
- **MVP 10 Reels.** Первый измеримый выход AI Growth Engine: 10 готовых Reels для публикации.

## 14. AI MEMORY MVP — УТВЕРЖДЁННЫЙ СЛЕДУЮЩИЙ ЭТАП

> Продуктовое решение владельца от 2026-07-08 (результат Owner Review). Реализация
> НЕ начата — этот раздел фиксирует спецификацию, не статус выполнения. Код
> пишется только по отдельному заданию на план реализации.

### Контекст решения

Стратегический этап OData (диагностика 2975 EntitySet, 1077 sales-кандидатов —
см. раздел 7) завершён. Вместо полного перехода каталога/остатков/цен на OData
первой практической интеграцией утверждена узкая функция: **AI Memory** — умение
Алины узнавать клиента и использовать историю его покупок из 1С для
персонализации. Это не новая архитектура и не Data Layer — это бизнес-функция
AI Seller поверх уже существующего механизма `<customer_context>` (раздел 4).

### Цель

Научить Алину узнавать клиента и использовать историю его покупок из 1С/OData.

**Телефон — основной бизнес-идентификатор клиента** внутри AI Factory для всех
каналов: Telegram, WhatsApp, сайт, мобильное приложение, будущие каналы.
Внутренний идентификатор контрагента в 1С — техническая деталь реализации.

### Product Specification — минимальный MVP

1. Получать из 1С/OData клиента по телефону.
2. Получать документы реализации клиента.
3. Получать строки товаров из этих документов.
4. На стороне AI Factory вычислять:
   - `total_orders`
   - `last_order_date`
   - `favorite_categories` — только при ≥2 заказах
   - `last_products`
   - `order_frequency`
   - `segment`: 0 → `new`, 1–5 → `regular`, 6–15 → `loyal`, 16+ → `vip`
5. Передавать эти данные в уже существующий блок `<customer_context>` (раздел 4;
   структура блока не меняется, SYSTEM_PROMPT.md не переписывается).
6. Использовать профиль для персонализации, рекомендаций и допродаж.

### Правила MVP — границы, за которые НЕ выходить

- Не строить Data Layer.
- Не делать полную синхронизацию 1С.
- Не подключать остатки, цены, себестоимость и аналитику на этом этапе.
- Не делать большой рефакторинг.
- Не ломать текущего работающего AI Seller (MVP — живой продакшн, FROZEN).
- Не обращаться к внешним консультантам.
- Не возвращаться к новым исследованиям OData — использовать уже найденные
  сущности (`odata_probe/`, `odata_sales_probe/`, раздел 7).

### Важные правила реализации

1. Телефон нормализуется перед поиском.
2. Если найдено несколько контрагентов с одним телефоном — выбирается контрагент
   с самой поздней датой последнего документа.
3. Если 1С/OData недоступна — диалог продолжается как сейчас, без профиля; ошибка
   логируется, клиенту не показывается (тот же паттерн graceful degradation, что
   и решение #20 для уведомления менеджера).
4. Профиль запрашивается один раз за сессию.
5. Приоритет имени: клиент сам назвал в диалоге → имя из 1С → имя из Telegram
   НЕ считается надёжным источником.
6. 1С — System of Record: факты покупок (решение #23).
7. AI Factory — System of Intelligence: выводы, предпочтения, сегмент, контекст
   общения (решение #23).

### ✅ Исследование завершено 2026-07-08 — Product Specification подтверждена без изменений

Полный аудит цифровой экосистемы клиента в 1С (интернет-магазин, Kaspi, Halyk,
Ozon, Wolt, розница, программа лояльности, возвраты) завершён. **Архитектура и
Product Specification выше НЕ меняются.** Уточнён только состав источников
данных для функции поиска клиента на Этапе 1 (решение #27):

- `Document_Kaspi_ЗаказыКлиентов` — основной источник (телефон/имя/адрес/сумма/товары встроены в документ, без join)
- `Document_Halyk_ЗаказыКлиентов` — дополнительный источник (решение #26; если есть данные)
- `Catalog_iq_ПользователиИнтернетМагазина` — пользователи сайта (телефон/email/имя/фамилия из регистрации)
- `Catalog_Контрагенты` — универсальный fallback для розницы/опта (телефон через `КонтактнаяИнформация`)

**Подтверждённые находки аудита:**
- Значительная часть Customer Intelligence уже много лет хранится внутри 1С (решение #25) — не нужно создавать новую клиентскую базу.
- Телефон подтверждён как основной бизнес-идентификатор клиента (решение #22) реальными данными: Kaspi/Halyk/сайт хранят телефон как первичный контакт.
- Источники клиентских данных подтверждены: интернет-магазин Rukodelie, Kaspi, Halyk (доп. источник), розничная торговля, программа лояльности, возвраты, скидки, история заказов, пользователи сайта.
- Ozon: интеграция есть, но клиентских данных не содержит (только логистика FBS). Wolt: интеграции нет.

**Главный вывод дня:** Главная задача AI Factory — объединить существующие данные
бизнеса в единый Customer Intelligence Profile, а не строить новую CRM.

**Да** — полноценную базу знаний о товарах для AI Seller можно построить почти полностью на

2. Характеристики (варианты) — `Catalog_ХарактеристикиНоменклатуры`.
3. Категории/иерархия — `Catalog_iq_Категории` (сайт, с SEO-полями) + внутренние папки номенклатуры.
4. Цвета/размеры/варианты — прямые поля номенклатуры + характеристики (два механизма).
5. Цены — `InformationRegister_ЦеныНоменклатуры`, 10 видов цен подтверждено (`Catalog_ВидыЦен`).
6. Остатки — `AccumulationRegister_СвободныеОстатки` (журнал движений, не витрина).
7. Остатки по складам — то же + `Catalog_Склады` (справочник, 59 полей).
8. Единицы измерения — `Catalog_УпаковкиЕдиницыИзмерения`, 5 параллельных измерений с коэффициентами.
9. Производитель/бренд — `Catalog_Производители` (чистый структурированный справочник с логотипами).
10. Связанные товары — `Catalog_iq_РекомендуемыеТовары` (одна связь «рекомендуем к X товар Y»).
11. Фотографии — `ФайлКартинки_Key` + `Catalog_НоменклатураПрисоединенныеФайлы` (ссылки подтверждены; бинарник через `$value` не проверен вживую).

Ни одной витрины «текущая цена»/«текущий остаток» (`_Balance`/`_SliceLast`) не опубликовано ни
для одной из 2975 сущностей — проверено явным поиском по всему списку. Цены и остатки доступны
только как журнал документов/движений. Получение «текущего» значения требует агрегации

- Не расширять `catalog_v1.csv` вручную, если нужные данные уже существуют в OData.
- OData — основной источник знаний о товарах для AI Product Knowledge.
- `catalog_v1.csv` — временный fallback/MVP-источник, не целевая архитектура.
