## 🧠 AI MEMORY MVP — УТВЕРЖДЁННЫЙ СЛЕДУЮЩИЙ ЭТАП

> Продуктовое решение владельца от 2026-07-08. Реализация НЕ начата — код по
> этому этапу пишется только отдельной командой после явного задания на план
> реализации. Этот раздел — зафиксированная спецификация, не статус выполнения.

### Цель

Научить Алину узнавать клиента и использовать историю его покупок из 1С/OData
для персонализации продаж (уже существующий `<customer_context>` XML-блок в
SYSTEM_PROMPT.md — см. PROJECT_MEMORY.md раздел 4 — начинает заполняться реальными
данными вместо пустого/статичного состояния).

**Телефон — основной бизнес-идентификатор клиента** внутри AI Factory для всех
каналов (Telegram, WhatsApp, сайт, мобильное приложение, будущие каналы).
Внутренний идентификатор контрагента в 1С — техническая деталь реализации, не
бизнес-идентификатор.

### Product Specification — минимальный MVP

1. Получать из 1С/OData клиента по телефону.
2. Получать документы реализации клиента.
3. Получать строки товаров из этих документов.
4. На стороне AI Factory вычислять:
   - `total_orders`
   - `last_order_date`
   - `favorite_categories` — только при ≥2 заказах
   - `last_products`
   - `order_frequency`
   - `segment`: 0 заказов → `new`, 1–5 → `regular`, 6–15 → `loyal`, 16+ → `vip`
5. Передавать эти данные в уже существующий блок `<customer_context>`.
6. Использовать профиль для персонализации, рекомендаций и допродаж.

### Правила MVP — границы, за которые НЕ выходить

| Запрет | Причина |
|--------|---------|
| ❌ Не строить Data Layer | MVP — узкая интеграция, не архитектурный слой |
| ❌ Не делать полную синхронизацию 1С | Только чтение по телефону, по запросу |
| ❌ Не подключать остатки, цены, себестоимость, аналитику на этом этапе | Вне скоупа AI Memory MVP |
| ❌ Не делать большой рефакторинг | Работающий AI Seller не трогаем |
| ❌ Не ломать текущего работающего AI Seller | MVP AI Seller — FROZEN, живой продакшн |
| ❌ Не обращаться к внешним консультантам | — |
| ❌ Не возвращаться к новым исследованиям OData | Используем уже найденные сущности (odata_probe/, odata_sales_probe/) |

### Важные правила реализации

1. **Телефон нормализуется** перед поиском.
2. Если найдено несколько контрагентов с одним телефоном — выбирается контрагент
   с **самой поздней датой последнего документа**.
3. Если 1С/OData недоступна — диалог продолжается как сейчас, **без профиля**;
   ошибка логируется, клиенту не показывается (тот же паттерн, что и с
   уведомлением менеджера — сбой внешней системы не блокирует основной сценарий).
4. Профиль запрашивается **один раз за сессию**.
5. **Приоритет имени:**
   - если клиент сам назвал имя в диалоге — использовать его;
   - иначе имя из 1С;
   - имя из Telegram **не считать надёжным источником**.
6. **1С — System of Record**: факты покупок.
7. **AI Factory — System of Intelligence**: выводы, предпочтения, сегмент, контекст общения.

### ✅ Исследование завершено 2026-07-08 — Product Specification подтверждена без изменений

Полный аудит цифровой экосистемы клиента в 1С (интернет-магазин, Kaspi, Halyk,

- `Document_Kaspi_ЗаказыКлиентов` — основной источник (телефон/имя/адрес/сумма/товары встроены в документ, без join)

Оба слоя — примеры одного и того же принципа (решение #23): **1С = System of Record, AI
Factory = System of Intelligence**. Не дублируют друг друга — AI Product Knowledge не хранит
ничего о клиенте, AI Memory не хранит ничего о товаре.
